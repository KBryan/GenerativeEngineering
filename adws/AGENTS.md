## Coding Rules

### Style

- Follow PEP 8, enforced by ruff
- Line length: 88 characters (black-compatible)
- Use double quotes for strings
- Use type hints on all function signatures

### Patterns

- **Models**: Define input/output schemas as separate Pydantic models (`TaskCreate`, `TaskUpdate`, `TaskResponse`). Never reuse input models for output.
- **Database**: All data access goes through `database.py` functions. Never read/write the in-memory dict directly from `app.py`.
- **Error handling**: Return `JSONResponse` with appropriate HTTP status codes (404 for not found, 422 for validation errors which FastAPI handles automatically).
- **Response model**: Always set `response_model` on route decorators for automatic validation and OpenAPI documentation.

### Testing

- One test file per source module
- Test file names: `test_<module>.py`
- Integration tests go in `tests/test_app.py`
- Unit tests for models go in `tests/test_models.py`
- Each test function tests one behavior
- Use FastAPI `TestClient` for API tests — no need to start a server
- Reset the database between tests using the `_clean_db` fixture

## Risk Areas

| Area | Risk | Mitigation |
|---|---|---|
| Data storage | In-memory dict — all data lost on restart | Document clearly; do not use in production |
| Authentication | None — API is fully open | Add auth middleware before exposing publicly |
| Input validation | Edge cases in priority values, title length | Pydantic validators + max-length constraints |
| Concurrency | Dict is not thread-safe | Acceptable for demo; use real DB for concurrency |
| ID generation | Simple integer counter — not UUID | Sufficient for demo; upgrade to UUID for production |
